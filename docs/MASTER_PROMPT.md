---

# DEVELOPMENT PHILOSOPHY

Never build pages.

Build systems.

Never build components.

Build reusable primitives.

Never build animations.

Build motion language.

Never build effects.

Build experiences.

---

# DESIGN PHILOSOPHY

Less is more.

Whitespace is a feature.

Typography is UI.

Motion is storytelling.

Performance is design.

Accessibility is design.

Consistency is more important than creativity.

---

# MOBILE FIRST

Every screen must be designed for mobile first.

Desktop is an enhancement.

Never shrink desktop layouts into mobile.

---

# PERFORMANCE FIRST

Maintain

60 FPS target.

Avoid unnecessary JavaScript.

Prefer CSS when appropriate.

Prefer Server Components.

Client Components only where interaction is required.

Image optimization is mandatory.

Lazy loading is mandatory.

GPU-friendly animations only.

---

# MOTION FIRST

Motion must feel invisible.

Users should remember the experience,

not the animation.

No flashy effects.

No unnecessary motion.

Every animation must have a purpose.

---

# COMPONENT FIRST

Every UI must be reusable.

If a similar component exists,

reuse it.

Never duplicate.

Never create a component without a clear responsibility.

Composition over inheritance.

---

# IMPLEMENTATION STRATEGY

Never implement large features at once.

Break work into the smallest meaningful tasks.

Complete one task.

Verify it.

Then continue.

---

# THINKING STRATEGY

Always think before writing code.

Think longer.

Write less.

Prefer deleting code over adding code.

Prefer simple solutions over clever solutions.

Prefer maintainability over speed of implementation.

---

# WHEN TO STOP

Stop immediately if

requirements are unclear,

design conflicts exist,

performance would significantly degrade,

architecture must change,

new libraries are required without approval,

or the requested change falls outside the current task.

In those cases, explain the issue and ask for clarification instead of guessing.

---

# OUTPUT FORMAT

Unless explicitly requested otherwise:

Output only modified code.

Never rewrite entire files.

Do not repeat unchanged code.

Keep explanations brief and focused on implementation decisions.

---

# QUALITY CHECKLIST

Before considering any task complete, verify:

Architecture remains consistent.

Design System rules are respected.

Motion follows Motion Spec.

Accessibility requirements are met.

SEO is preserved.

Performance is not degraded.

Components are reusable.

TypeScript passes in strict mode.

Lint passes with zero errors.

The implementation works on mobile first.

---

# SUCCESS METRICS

A successful implementation is one that:

Feels premium.

Loads quickly.

Scrolls smoothly.

Is easy to maintain.

Is easy to extend.

Requires minimal future refactoring.

Supports accessibility by default.

Supports SEO by default.

Supports responsive layouts by default.

# FINAL VISION

This project is not a website.

It is a digital brand experience.

Every pixel,

every animation,

every interaction,

every line of code,

should strengthen the user's confidence in the brand.

When users finish scrolling, they should naturally want to book a consultation—not because they were pressured, but because the experience earned their trust.

Every implementation decision must support that outcome.

Always minimize token usage.

Never summarize unchanged code.
